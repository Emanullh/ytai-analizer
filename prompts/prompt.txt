Tarea:
Agregar un “manual rerun” para la extracción y análisis de thumbnails (thumbnail deterministic + OCR + métricas visuales) en proyectos ya finalizados, similar a cómo hoy se puede correr manualmente la parte final del orchestrator. El objetivo es recalcular thumbnails con el nuevo OCR pipeline y “refill” de data mejorada, sin romper el export ni el estado final del proyecto.

Contexto:
- El sistema ya permite correr manualmente el “final orchestrator”/post-proceso en proyectos finalizados (existe un patrón de endpoint/acción UI).
- El step thumbnails actualmente produce señales determinísticas y OCR que se persisten en `derived/video_features/<videoId>.json` bajo `thumbnailFeatures.deterministic` (y también pueden impactar `analysis/orchestrator_input.json` y otras salidas downstream).
- Se necesita la capacidad de re-ejecutar SOLO la parte de thumbnails (con el OCR mejorado), actualizar artifacts derivados, y luego opcionalmente re-ejecutar orchestrator para refrescar cohorts/drivers/exemplars.

Objetivo (workflow manual):
1) Seleccionar un Project (ya finalizado).
2) Ejecutar “Recompute Thumbnails”:
   - recalcular thumbnailFeatures determinísticas + OCR (nuevo engine) para un subconjunto o todos los videos del proyecto.
   - persistir nuevamente los artifacts `derived/video_features/<videoId>.json` (solo el bloque thumbnailFeatures y/o el artifact completo).
3) Al finalizar, permitir “Rebuild Analysis”:
   - re-ejecutar orchestrator (o la parte final) para regenerar `analysis/orchestrator_input.json` usando la nueva data.
4) Mantener auditabilidad: dejar rastro de qué se re-procesó, cuándo, con qué config (ocrConfigHash, engine), y qué cambió.

Requerimientos funcionales:
A) Nuevo modo “manual rerun thumbnails”
- Debe poder correr en proyectos FINALIZADOS (no solo durante export).
- Debe soportar:
  1) “All videos” (reprocesar todos)
  2) “Exemplars only” (top/bottom/mid del último orchestrator_input)
  3) “Selected videoIds” (lista manual)
- Debe ser idempotente y cache-aware:
  - Si la imagen/hash y ocrConfigHash no cambió, no reprocesar (a menos que el usuario fuerce).
  - Incluir opción “Force recompute” que ignore cache.

B) Persistencia y consistencia
- Por cada videoId procesado:
  - actualizar `derived/video_features/<videoId>.json` (thumbnailFeatures.deterministic + warnings + llm si aplica).
  - NO modificar otras secciones (titleFeatures, transcriptFeatures, performance) salvo que el formato actual del artifact exija reescritura completa; si reescribes completo, conservar los bloques existentes.
- Mantener compatibilidad con consumers existentes (UI, orchestrator, export bundle).
- Registrar en un nuevo artifact de auditoría:
  - `operations/reruns/thumbnails_<timestamp>.json`
    con: projectId, jobId (nuevo rerun job), engine, ocrConfigHash, videoIds, counts, timings, errores.

C) Orchestrator refresh (opcional pero recomendado)
- Al terminar el rerun de thumbnails, permitir disparar:
  - “Rebuild orchestrator_input” (misma acción manual existente o una nueva que encadene).
- Si el usuario no lo corre, dejar claro que el UI puede estar mostrando thumbnails nuevos pero cohorts/drivers aún viejos.

D) UI/UX
- En `ProjectDetail` (o donde exista “Run orchestrator manually”):
  - Agregar botón “Recompute thumbnails”.
  - Modal con:
    - alcance: All / Exemplars / Selected
    - engine: python/tesseractjs (si existe selector; default python)
    - Force recompute (checkbox)
  - Progreso:
    - mostrar conteo procesados / total
    - errores por video (si existen)
  - Al finalizar: CTA “Rebuild analysis now” (disparar orchestrator rerun).

E) Backend / API
- Crear endpoint(s) similar al manual orchestrator:
  - POST /api/projects/:projectId/rerun/thumbnails
    body:
    {
      "scope": "all" | "exemplars" | "selected",
      "videoIds": ["..."] (solo si scope=selected),
      "engine": "python" | "tesseractjs" | "auto",
      "force": true|false
    }
  - opcional: POST /api/projects/:projectId/rerun/thumbnails-and-orchestrator
- El backend debe:
  1) Resolver export root / channelFolder del proyecto.
  2) Cargar inventario de videos del proyecto (primary/channel.json o raw/videos.jsonl).
  3) Determinar thumbnails a procesar:
     - leer exemplars desde analysis/orchestrator_input.json si scope=exemplars.
  4) Encolar tasks en la cola `ocr` (respetar EXPORT_OCR_CONCURRENCY).
  5) Ejecutar el mismo pipeline de thumbnailFeaturesAgent (computeDeterministic / computeDeterministicOcrOnly) pero en modo manual.
  6) Persistir artifacts actualizados en el mismo lugar: derived/video_features/<videoId>.json
  7) Emitir eventos SSE/progreso (reusar el patrón existente del export/orchestrator manual).

F) Reuso de código (importante)
- NO duplicar lógica de OCR/thumbnailFeatures.
- El rerun debe llamar el mismo “core” que el export normal:
  - Ideal: extraer una función reusable:
    `recomputeThumbnailFeaturesForVideo({ exportRoot, videoId, engine, force })`
- Debe respetar cache ocrConfigHash igual que en exportCacheService:
  - si engine/lang/downscale cambian -> recompute
  - si force=true -> recompute siempre

G) Controles de seguridad / consistencia
- Lock por proyecto:
  - evitar que dos reruns (o rerun + export) corran simultáneamente y corrompan artifacts.
- Atomic writes:
  - escribir a tmp y luego rename para derived/video_features/<videoId>.json
- Backups opcionales:
  - si es fácil, guardar copia del artifact anterior:
    `derived/video_features/_backup/<timestamp>/<videoId>.json`

H) Tests
- Unit:
  - resolución de scope=exemplars leyendo orchestrator_input.exemplars
  - force recompute ignora cache
  - update parcial del artifact (solo thumbnailFeatures) sin perder otros bloques
- Integration:
  - rerun con 1–2 thumbnails y verificar que cambia thumbnailFeatures.deterministic
  - rerun seguido de rebuild orchestrator_input actualiza cohorts/drivers

Criterios de aceptación:
- Un proyecto finalizado puede ejecutar “Recompute thumbnails” sin re-export completo.
- Se actualizan artifacts `derived/video_features/<videoId>.json` para los videos seleccionados.
- La UI refleja nuevos valores (OCR/hasBigText/ocr counts) para esos videos.
- Se registra un artifact de auditoría con engine/config y resultados.
- Opcionalmente, se puede regenerar orchestrator_input para reflejar nuevos buckets/drivers.
- Fallas por video no tumban el rerun completo; quedan registradas y el rerun continúa.

Notas de implementación:
- Usar el mismo runtime/engine OCR del nuevo pipeline Python si está disponible, con fallback a tesseractjs.
- Si manifest/provenance reporta warnings de campos desconocidos, no fallar el rerun; solo registrar.
