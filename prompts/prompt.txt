Tarea: Implementar logging/tracing completo (event log + error log) para export jobs con trazabilidad end-to-end

Contexto:
- Export flow asíncrono con SSE:
  - POST /api/export/jobs -> jobId
  - GET /api/export/jobs/:jobId/events -> stream SSE + replay
- Backend coordina en:
  - apps/api/src/server.ts
  - apps/api/src/services/exportJobService.ts
  - apps/api/src/services/exportService.ts
- Side effects: filesystem bajo exports/<channel>/...
- Hay múltiples sub-sistemas con riesgo de fallo:
  - YouTube API, transcriptPipeline (captions/ASR), ASR worker, AutoGen worker, OCR, embeddings, cache service, orquestador
- Requisito: “saber en qué momento falla” y poder reproducir/debuggear después, sin cambiar UX ni agregar endpoints nuevos.

Objetivo:
Agregar un sistema de logs estructurados y persistentes por job, con:
1) Event timeline (todas las etapas, timings, inputs clave, decisiones cache hit/miss, etc.)
2) Error log (errores con stacktrace, categoría, contexto, correlación con step/video)
3) Correlation IDs (jobId + videoId + stepId + requestId) para unir eventos.
4) Persistencia en disco:
   exports/<channel>/logs/
     job_<jobId>.events.jsonl
     job_<jobId>.errors.jsonl
     job_<jobId>.summary.json
   y/o dentro de exports/<channel>/analysis/ si prefieres, pero debe ser consistente y documentado.

No romper contratos:
- No cambiar SSE event names obligatoriamente.
- Se permite EMITIR warnings SSE adicionales (ya existe "warning") para reflejar fallos y que el usuario vea algo.
- No agregar pasos extra en frontend.

Diseño (recomendado):
A) Logger estructurado por job (JSONL)
- JSONL por performance y append-only.
- Cada línea: 1 evento.
- Campos comunes obligatorios:
  - ts (ISO)
  - level: trace|debug|info|warn|error
  - jobId
  - stepId (uuid corto o incremental)
  - scope: "exportJob"|"exportService"|"youtube"|"transcript"|"asr"|"autogen"|"ocr"|"embeddings"|"cache"|"orchestrator"|"fs"
  - action: string (ej "video_start", "cache_hit", "download_thumbnail", "asr_request", ...)
  - videoId (opcional)
  - stage (opcional, de los stages actuales)
  - msg (string corto)
  - data (object pequeño, redaccionado)
- Para errores:
  - error: { name, message, stack, code?, cause? }
  - retry: { attempt, max, willRetry }

B) Redacción / privacidad (no filtrar secretos)
- Nunca loggear:
  - API keys (YOUTUBE_API_KEY, OPENAI_API_KEY)
  - prompts completos del LLM (puedes loggear hash + tamaño)
- Sí loggear:
  - modelos (AUTO_GEN_MODEL_*, embedding model)
  - flags enabled/disabled
  - sizes/counts/timings
  - file paths RELATIVOS (no absolutos) bajo exports/<channel>/...

C) Summary final por job
- Escribir job_<jobId>.summary.json al finalizar (done/failed):
  - startedAt, finishedAt, durationMs
  - status done|failed
  - exportedCount
  - warningsCount, errorsCount
  - perStageTimingsMs (aggregate)
  - perVideo: status + timings + cacheHit type + transcriptStatus + llmUsed flags
  - lastError (si failed) con scope/action/videoId

Implementación requerida:
1) Crear módulo base:
- apps/api/src/observability/jobLogger.ts
  - createJobLogger({exportRootAbs, channelFolder, jobId}): JobLogger
  - logger.event({level, scope, action, stage?, videoId?, msg?, data?})
  - logger.error({scope, action, stage?, videoId?, err, data?, retry?})
  - logger.flush() (opcional)
  - logger.close()
  - logger.summary({status, ...})

- Debe abrir 2 write streams:
  - events.jsonl
  - errors.jsonl
  append-only, fsync razonable (flush en puntos clave).

2) Integración en exportJobService (alto nivel):
- Al crear job: log event "job_created"
- Al iniciar running: "job_started" + snapshot config (sin secretos):
  - exportVersion, timeframe, AUTO_GEN_ENABLED, LOCAL_ASR_ENABLED, THUMB_OCR_ENABLED, embeddingModel, llmModels, cacheEnabled
- Al finalizar: "job_done"/"job_failed" con summary + escribir summary.json

3) Integración en exportService (bajo nivel, por video):
Añadir eventos en puntos críticos:
- video_start (con videoId)
- cache_check_start / cache_hit_full / cache_hit_partial / cache_miss
  - data: reasons[], invalidationFlags, willCompute:{...}
- transcript_start / captions_attempt / captions_result / asr_attempt / asr_result
  - data: transcriptSource, transcriptStatus, segmentsCount, transcriptLen
- thumbnail_download_start / thumbnail_download_done
  - data: urlDomain, bytes, ms
- ocr_start / ocr_done (si THUMB_OCR_ENABLED)
  - data: ocrWordCount, textAreaRatio, ms
- embeddings_start / embeddings_done (si OPENAI_API_KEY)
  - data: model, inputChars, ms
- autogen_task_start / autogen_task_done (por task: description/transcript/thumbnail/orchestrator)
  - data: taskName, model, reasoningEffort, inputBytes, outputBytes, ms, ok
- derived_write_start / derived_write_done
- raw_write_start / raw_write_done
- orchestrator_start / orchestrator_done
- video_done (con timings agregados)
- En cada catch:
  - logger.error(...) con err + scope + action + stage + videoId + retry info
  - emitir SSE warning (ya existe) con message corto y correlacionable:
    "ERR <scope>/<action> stepId=<...> (see logs/job_<jobId>.errors.jsonl)"

4) Correlation IDs y stepIds
- Implementar helper:
  - apps/api/src/observability/ids.ts
    - newStepId()
- Cada operación relevante genera stepId y se propaga (en data/event).
- Cuando llames workers (ASR/AutoGen) ya existe id uuid:
  - loggear mapping: workerRequestId <-> stepId.

5) Manejo de errores y clasificación
- Implementar classifyError(err) -> { code, kind }:
  - network_timeout, rate_limited, invalid_response, fs_error, worker_crash, schema_validation_failed, etc.
- Guardar code/kind en error log para filtros.

6) “Debug bundle” opcional (sin secretos)
- En caso de job_failed, crear:
  exports/<channel>/logs/job_<jobId>.debug_bundle.json
  con:
  - manifest/artifacts list
  - last 200 events
  - last 50 errors
  - env snapshot sin secretos
  Esto acelera reporte de bugs.
