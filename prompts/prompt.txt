Objetivo UX:
1) Agregar un dashboard adicional “Projects” donde el usuario pueda:
   - ver lista de proyectos/export runs organizados por “proyecto” (carpeta en exports/<channel>)
   - abrir un proyecto y visualizar:
     - summary del job más reciente (logs/job_<jobId>.summary.json)

     - playbook.json y templates.json en vistas legibles
     - lista de videos con thumbnails + performance + flags (transcriptStatus, cache hits si aplica)
     - botón para abrir detalles de un video (derived/video_features/<videoId>.json) y transcript (raw/transcripts/<videoId>.jsonl) en vista expandible
2) Mantener el flujo actual:
   - /analyze + export SSE siguen igual
   - solo agregamos una nueva sección de UI (navegación)
3) Mejorar UI actual con Tailwind:
   - layout consistente (header/nav, cards, tables, badges)
   - modal de progreso más claro y visual (sin cambiar lógica del reducer)

Definición “Project”:
- Un “Project” = una carpeta bajo exports/<channel_sanitizado>/ (por canal).
- Dentro puede haber múltiples jobs (logs/job_<jobId>.*). En el dashboard:
  - el “project list item” muestra el último job (por exportedAt o por logs summary timestamp)
  - el detalle muestra lista de jobs (opcional MVP) y permite ver logs por job.

Requisitos backend (apps/api):
A) Implementar endpoints de lectura “read-only” (NO nuevos writes):
1) GET /projects
   Respuesta: lista de proyectos:
   [
     {
       "projectId": "Canal_Demo",                 // channelFolder
       "channelId": "...",
       "channelName": "...",
       "exportVersion": "1.1",
       "lastExportedAt": "ISO",
       "lastJobId": "uuid|null",
       "counts": { ... }                          // desde manifest si existe
       "warningsCount": number,
       "status": "ok|partial|failed|unknown"      // derivado del último summary o manifest
     }
   ]
   Implementación:
   - enumerar directorios bajo exports/
   - por cada uno, leer manifest.json si existe (y channel.json como fallback)
   - determinar lastJobId:
     - preferir logs/job_*.summary.json más reciente
     - si no hay logs, null
   - NO bloquear si un project está corrupto: incluirlo con status unknown y warning.

2) GET /projects/:projectId
   Respuesta: detalle de proyecto:
   {
     "projectId": "...",
     "channel": { channelId, channelName, sourceInput, timeframe, exportedAt, timeframeResolved },
     "manifest": {...}|null,
     "latestJob": {
       "jobId": "...",
       "status": "done|failed",
       "startedAt": "ISO",
       "finishedAt": "ISO",
       "durationMs": number,
       "warningsCount": number,
       "errorsCount": number
     }|null,
     "jobs": [
       { jobId, status, startedAt, finishedAt, durationMs, warningsCount, errorsCount }
     ],
     "artifacts": {
       "playbook": "analysis/playbook.json"|null,
       "templates": "derived/templates.json"|null,
       "channelModels": "derived/channel_models.json"|null
     }
   }
   - jobs[]: listar logs/job_*.summary.json (orden desc por finishedAt)
   - artifacts: verificar existencia de archivos

3) GET /projects/:projectId/videos
   Respuesta: lista de videos (ligera, sin transcript full):
   [
     {
       "videoId": "...",
       "title": "...",
       "publishedAt": "...",
       "thumbnailPath": "thumbnails/<id>.jpg",
       "transcriptStatus": "ok|missing|error",
       "transcriptSource": "captions|asr|none",
       "performance": { viewsPerDay, engagementRate, residual, percentile }|null,
       "hasLLM": { description:boolean, transcript:boolean, thumbnail:boolean } // derivado de derived/video_features
     }
   ]
   Implementación:
   - leer channel.json (videos base + transcriptStatus/source)
   - para cada videoId, leer derived/video_features/<id>.json si existe:
     - extraer performance + flags de llm presence
   - optimización: leer en paralelo con límite (usar EXPORT_FS_CONCURRENCY o un p-limit interno).

4) GET /projects/:projectId/videos/:videoId
   Respuesta: detalles completos:
   {
     "videoId": "...",
     "derived": {...}|null,                 // derived/video_features/<id>.json completo
     "transcriptJsonl": [ ... ]|null,       // parsear raw/transcripts/<id>.jsonl y devolver como array de objetos
     "rawVideo": {...}|null                 // opcional: buscar en raw/videos.jsonl la línea del videoId
   }
   Nota: transcript puede ser grande; si preocupa, implementar query param ?maxSegments=200 o ?truncateChars=... (MVP opcional).

5) GET /projects/:projectId/artifacts/playbook
   GET /projects/:projectId/artifacts/templates
   GET /projects/:projectId/artifacts/channel_models
   - Devuelven JSON directo para consumo en UI.
   - Alternativa: usar el GET /projects/:projectId y luego fetch a rutas específicas.
