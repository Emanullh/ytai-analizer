Tarea: Paralelizar el pipeline de export de forma segura (concurrency configurable por ENV) + validación de qué tareas pueden correr en paralelo

Contexto:
- Export job asíncrono con SSE en apps/api/src/services/exportJobService.ts + exportService.ts
- Por video se ejecuta un pipeline con steps típicos:
  - (A) cache_check
  - (B) transcript (captions + fallback ASR local)
  - (C) download thumbnail
  - (D) derived features:
      - title (det + embeddings opcionales + llm opcional)
      - description (det + llm opcional)
      - transcript (det + embeddings opcionales + llm opcional)
      - thumbnail (det + OCR opcional + llm opcional)
  - (E) write raw + derived JSON
- Luego cross-video (siempre fresh):
  - performanceNormalization (determinista)
  - orchestratorDeterministic + orchestratorService (LLM opcional)
- Requisito: identificar qué es paralelizable y ejecutarlo concurrentemente.
- La cantidad de concurrent tasks debe definirse por env.

Objetivo:
1) Implementar un “Task Scheduler” interno con concurrencia configurable por env:
   - concurrencia por “tipo de tarea” y/o global
2) Paralelizar:
   - Entre videos: procesar múltiples videos en paralelo (con límites)
   - Dentro de un video: paralelizar steps independientes (con límites) sin romper dependencias
3) Mantener compatibilidad:
   - mismos endpoints, mismo flujo UI
   - SSE stages existentes (queue, transcribing, downloading_thumbnail, writing_json, etc.)
4) No saturar recursos:
   - ASR local en GPU debe seguir con concurrency bajo (default 1)
   - OCR puede ser pesado (default 1-2)
   - LLM/embeddings deben limitarse (default 2-4)
   - Descargas HTTP (thumbnail/captions/youtube api) puede ser mayor (default 4-8)
5) Debe existir validación explícita (documentada y testeada) de dependencias por step.

Nuevas variables ENV (apps/api/.env.example + env.ts):
- EXPORT_VIDEO_CONCURRENCY (default 3)                 // cuántos videos simultáneos
- EXPORT_HTTP_CONCURRENCY (default 6)                   // descargas/YouTube API/captions
- EXPORT_ASR_CONCURRENCY (default 1)                    // ya existe LOCAL_ASR_MAX_CONCURRENCY, usarlo o mapear
- EXPORT_OCR_CONCURRENCY (default 2)
- EXPORT_LLM_CONCURRENCY (default 2)
- EXPORT_EMBEDDINGS_CONCURRENCY (default 2)
- EXPORT_FS_CONCURRENCY (default 6)                     // writes (JSON/thumbnails), puede ser 1 si prefieres
- EXPORT_FAIL_FAST (default false)                      // si true, abortar job ante el primer error (hoy debe ser false)

Reglas de paralelización (debe quedar en código + docs):
A) Dependencias por video (DAG):
- transcript -> transcript-derived (transcript features) requiere transcript
- thumbnail download -> thumbnail-derived requiere thumbnail local
- title-derived requiere title + (opcional) transcript para coverage; pero si transcript no listo aún:
  - computar title deterministic parcial (sin coverage) y completar luego cuando transcript esté listo (upgrade)
  - o declarar dependencia titleDerived.waitForTranscript=true (decidir una sola estrategia y documentar)
- description-derived requiere description (ya se obtiene de raw/videos.jsonl o YouTube metadata)
- write raw transcripts requiere transcript result
- write derived puede hacerse incremental cuando cada sub-feature esté lista (merge)
B) Entre videos:
- Los videos son independientes excepto por recursos compartidos (rate limits, GPU, OCR).
- performanceNormalization + orchestrator deben correr solo cuando todos los videos del subset hayan terminado.

Implementación requerida:
1) Introducir un scheduler ligero (sin dependencias pesadas):
- Preferido: usar `p-limit` y colas por tipo.
  - Crear apps/api/src/services/taskScheduler.ts
    - createScheduler({video, http, asr, ocr, llm, embeddings, fs}): Scheduler
    - scheduler.run("http", fn) / scheduler.run("llm", fn) etc.
    - scheduler.runVideo(videoId, fn) aplica límite de videos
  - Alternativa: implementación propia con semáforos (Promise queue).
2) Modificar exportService.ts para:
- Procesar videos usando scheduler.runVideo(videoId, async ()=> pipelineVideo(videoId))
- Dentro de pipelineVideo:
  - arrancar tareas paralelas cuando sea seguro:
    a) cache_check (primero, sync)
    b) en paralelo (según cache):
       - transcript: scheduler.run("http" + "asr") ya interno
       - thumbnail download: scheduler.run("http")
       - description deterministic: inmediato
       - title deterministic básico: inmediato
    c) cuando transcript termina:
       - title coverage + transcriptFeatures deterministic: scheduler.run("embeddings" opcional)
       - transcript LLM (autogen): scheduler.run("llm") (si enabled)
       - escribir raw transcript JSONL: scheduler.run("fs")
    d) cuando thumbnail termina:
       - OCR: scheduler.run("ocr") (si enabled)
       - thumbnail LLM (autogen multimodal): scheduler.run("llm") (si enabled)
       - escribir thumbnail + update cache: scheduler.run("fs")
    e) derived merge/writes:
       - cada sub-feature al completarse hace merge incremental y write atómico via scheduler.run("fs")
- Debe quedar una función clara que defina el DAG (documentarla en comments).
3) Integración con Export Cache:
- Si cache hit full:
  - pipelineVideo solo “revalida existencia” y emite eventos mínimos (fs checks bajo scheduler.fs)
- Si cache partial:
  - solo schedule de tasks faltantes según invalidación granular
4) SSE/Progress:
- Mantener stages existentes:
  - transcribing -> solo cuando transcript task esté ejecutándose
  - downloading_thumbnail -> cuando download thumbnail está ejecutándose
  - writing_json -> cuando hay writes/merge/orchestrator
- Para paralelismo:
  - Asegurar que video_progress percent y stage no se contradigan:
    - si transcript y thumbnail corren a la vez, elegir stage por prioridad:
      transcribing > downloading_thumbnail > writing_json
    - o emitir stage según “último cambio relevante” (documentar)
- Emite warnings si se detecta “backpressure” (colas largas) opcional.
